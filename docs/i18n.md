# System i18n

## Zakres

Po Fazie 2 projekt ma aktywne PL/EN, ale jest to i18n zorientowane na klienta. Jezyk nie jest jeszcze czescia routingu ani nie jest przechowywany w cookie po stronie serwera.

## Architektura

System ma dwa zrodla tlumaczen:

| Warstwa | Zrodlo | Zastosowanie |
|---------|--------|--------------|
| Statyczne UI | `messages/pl.json`, `messages/en.json` | menu, teksty pomocnicze, czesc etykiet |
| Dynamiczna tresc | `Section.contentPl`, `Section.contentEn` | copy edytowane przez admina |

## Provider i hook

Najwazniejszy plik: `lib/i18n.ts`.

Udostepnia:

| Element | Rola |
|--------|------|
| `I18nProvider` | trzyma aktywny jezyk i wystawia kontekst |
| `useLocale()` | daje dostep do `locale`, `setLocale()` i `t()` |
| `t(key)` | czyta tlumaczenie z pliku JSON lub zwraca sam klucz, gdy brak wpisu |

Przykladowe uzycie:

```tsx
import { useLocale } from '@/lib/i18n';

export function Example() {
  const { locale, setLocale, t } = useLocale();

  return (
    <button onClick={() => setLocale(locale === 'pl' ? 'en' : 'pl')}>
      {t('nav.openMenu')}
    </button>
  );
}
```

## Zrodlo prawdy dla jezyka

### Aktualny mechanizm

- domyslny jezyk w providerze to `pl`,
- przy starcie klient sprawdza `localStorage` pod kluczem `hommm_locale`,
- `setLocale()` zapisuje nowa wartosc do `localStorage`,
- `setLocale()` aktualizuje tez `document.documentElement.lang`.

### Wazny niuans

`app/layout.tsx` renderuje HTML z `lang=\"pl\"`. Jezeli uzytkownik mial zapisane `en` w `localStorage`, to przejscie na angielski nastapi dopiero po hydracji klienta. To oznacza:

- brak pelnego SSR per locale,
- potencjalny chwilowy mismatch jezyka na pierwszym renderze.

## Jak frontend wybiera tresc sekcji

W `HomeClient.tsx` helper `c(section, field)` wybiera:

- `contentPl[field]`, gdy `locale === 'pl'`,
- `contentEn[field]`, gdy `locale === 'en'`.

Dla sekcji rezerwacji jest jeszcze helper `r(field)`, ktory:

1. najpierw czyta tekst z sekcji `rezerwacja` w DB,
2. dopiero potem probuje fallback do `t('reservation.<field>')`.

To jest wazne, bo w obecnej implementacji sekcja rezerwacji jest glownie sterowana przez dane z CMS.

## Pliki z tlumaczeniami statycznymi

Format jest plaski:

```json
{
  "menu.koncept": "koncept",
  "section.readMore": "CZYTAJ WIECEJ",
  "footer.contact": "KONTAKT:"
}
```

Konwencja kluczy:

- `menu.*`
- `reservation.*`
- `section.*`
- `footer.*`
- `nav.*`

## Co jest tlumaczone dzisiaj

| Obszar | Zrodlo |
|--------|--------|
| Top menu | pliki `messages/*.json` |
| CTA "Czytaj wiecej" | pliki `messages/*.json` |
| Czesc stopki | pliki `messages/*.json` + dane kontaktowe z CMS |
| Tresc sekcji `hero`, `koncept`, `miejsce`, `rezerwacja`, `kontakt` | DB / fallback content |

## Ograniczenia obecnego i18n

| Obszar | Ograniczenie |
|--------|--------------|
| Routing | brak `/en/...` i brak jezyka w URL |
| SSR | poczatkowy HTML zawsze ma `lang=\"pl\"` |
| Trwalosc po stronie serwera | brak cookie z jezykiem |
| Pokrycie kluczy | pliki `messages` nie sa zrodlem 1:1 dla calej sekcji rezerwacji; tam priorytet ma CMS |
| Admin panel | sam panel admina pozostaje glownie po polsku |

## Jak dodac nowy jezyk

1. dodaj nowy plik `messages/<lang>.json`,
2. rozszerz typ `Locale` w `lib/i18n.ts`,
3. dodaj plik do `messagesMap`,
4. dodaj przelacznik w `components/TopMenu.tsx`,
5. uzupelnij odpowiednie pola tresci w CMS albo rozbuduj model, jesli potrzebujesz wiecej niz PL/EN.

## Wnioski

Obecny system i18n jest wystarczajacy dla jednej strony i dwoch jezykow. Jego najmocniejsza strona to prostota i integracja z CMS. Najwieksze ograniczenie to brak pelnego server-side locale handling, co warto rozwazyc dopiero wtedy, gdy dojdzie routing wielostronicowy lub wymagania SEO dla wielu jezykow.
